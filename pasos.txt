PASOS DEL PROYECTO — TFM
Asistente conversacional NL → MQL para MongoDB
================================================

Basado en el anteproyecto (Marzo 2026). El desarrollo sigue un enfoque iterativo
dividido en dos fases principales + evaluación final.

─────────────────────────────────────────────
FASE 0 — PREPARACIÓN DEL ENTORNO
─────────────────────────────────────────────

[ ] 0.1  Crear y activar entorno virtual Python (python -m venv .venv)
[ ] 0.2  Instalar dependencias iniciales (pip install -r requirements.txt)
[ ] 0.3  Configurar .env con claves de API (OpenAI o Anthropic) y URI de MongoDB
[ ] 0.4  Levantar instancia MongoDB local o en Atlas para desarrollo
[ ] 0.5  Elegir el LLM inicial para la Fase 1:
         - GPT-4o / GPT-4-turbo  → mayor precisión, coste por token
         - Claude 3.5 Sonnet     → buen balance coste/calidad
         - LLaMA 3 (local)       → privacidad máxima, requiere GPU
         Criterios: precisión en generación de código, coste, latencia, privacidad.
[ ] 0.6  Elegir framework web: FastAPI (recomendado, async nativo) vs Flask (más simple)
[ ] 0.7  Elegir orquestador LLM: LangChain vs LlamaIndex
         - LangChain: más maduro, más ejemplos de text-to-query
         - LlamaIndex: mejor para RAG sobre documentos/esquemas


─────────────────────────────────────────────
FASE 1 — PROTOTIPO CON ESQUEMA FIJO
─────────────────────────────────────────────
Objetivo: validar el flujo completo sobre una sola colección MongoDB con
esquema conocido. No se requiere generalización todavía.

[ ] 1.1  Seleccionar o crear el dataset de prueba:
         - Opción A: dataset público (ej. sample_mflix de MongoDB Atlas,
                     Northwind portado a MongoDB, dataset de e-commerce)
         - Opción B: dataset sintético con patrones analíticos habituales
                     (ventas, pedidos, usuarios, productos)
         El dataset debe cubrir: filtros simples, comparaciones, agrupaciones,
         $lookup entre colecciones, arrays anidados.

[ ] 1.2  Definir el esquema fijo en data/schemas/ (JSON con campos, tipos,
         ejemplos de valores). Este fichero se inyectará en cada prompt.

[ ] 1.3  Implementar src/core/db_connector.py
         - Conexión a MongoDB vía pymongo
         - Método execute_query(collection, pipeline_or_filter) → lista de docs
         - Gestión de errores de conexión y de ejecución de queries

[ ] 1.4  Diseñar las plantillas de few-shot prompting en src/prompts/
         Cada plantilla debe incluir:
           * Descripción del rol del LLM ("eres un experto en MQL...")
           * Esquema completo de la colección (campos + tipos + ejemplos)
           * 5-10 pares (pregunta NL → MQL) representativos
           * Placeholder para la pregunta del usuario
         NOTA: la calidad de los few-shots es el factor más importante para
         la precisión del sistema. Dedicar tiempo a curarlos bien.

[ ] 1.5  Implementar src/core/mql_generator.py
         - Recibe: pregunta NL + nombre de colección
         - Carga la plantilla de prompts correspondiente
         - Llama al LLM y extrae el bloque MQL de la respuesta
         - Valida que la salida sea JSON parseable antes de ejecutar
         - Devuelve: query MQL (dict/list) + colección destino

[ ] 1.6  Implementar src/core/nlp.py (módulo de comprensión NL)
         - Detectar la colección objetivo a partir de la pregunta
         - Extraer entidades clave si es necesario para el prompt
         - En Fase 1 puede ser simple (colección única, sin enrutamiento)

[ ] 1.7  Construir el pipeline completo de extremo a extremo:
         pregunta → nlp → mql_generator → db_connector → resultado formateado

[ ] 1.8  Crear la interfaz web en src/web/app.py
         - Ruta POST /query: recibe pregunta, devuelve {mql, resultados}
         - Ruta GET /: interfaz HTML simple (caja de texto + tabla de resultados)
         - Mostrar la query MQL generada junto al resultado (transparencia)

[ ] 1.9  Pruebas manuales del prototipo Fase 1 con 20-30 preguntas variadas.
         Anotar fallos para mejorar los few-shots.


─────────────────────────────────────────────
FASE 2 — INFERENCIA DINÁMICA DE ESQUEMA
─────────────────────────────────────────────
Objetivo: que el sistema funcione con cualquier colección MongoDB sin
configuración manual del esquema.

[ ] 2.1  Implementar src/core/schema_inferrer.py
         - Samplear N documentos de la colección (ej. 50-100)
         - Inferir campos, tipos y ejemplos de valores automáticamente
         - Manejar campos opcionales y documentos anidados

[ ] 2.2  Integrar la inferencia dinámica en el pipeline de generación:
         si no existe esquema manual en data/schemas/, inferirlo en tiempo real.

[ ] 2.3  Implementar enrutamiento multi-colección:
         - Dado el listado de colecciones disponibles, el LLM (o una heurística)
           selecciona la colección más adecuada para cada pregunta.
         - Para preguntas que requieren $lookup entre colecciones, generar
           un aggregation pipeline completo.

[ ] 2.4  Añadir memoria conversacional (historial de la sesión):
         - Permitir preguntas de seguimiento ("¿y en febrero?", "¿solo los activos?")
         - LangChain ConversationBufferMemory o equivalente

[ ] 2.5  Pruebas con colecciones no vistas durante el desarrollo.


─────────────────────────────────────────────
EVALUACIÓN
─────────────────────────────────────────────

[ ] 3.1  Construir el benchmark en data/benchmark/
         - Mínimo 50 pares (pregunta NL, MQL esperado) por colección
         - Cubrir: filtros simples, operadores ($gt,$lt,$in), agrupaciones
           ($group, $sum, $avg), ordenación, límite, $lookup, arrays ($elemMatch)
         - Incluir preguntas ambiguas o con errores tipográficos

[ ] 3.2  Implementar el script de evaluación (tests/eval.py):
         - Ejecutar ambas queries (generada vs esperada) sobre los mismos datos
         - Comparar resultados (no el texto MQL, sino el resultado devuelto)
         - Calcular: Exactitud funcional (%), consultas parcialmente correctas,
           tasa de error de ejecución (MQL inválido)

[ ] 3.3  Comparativa de modelos LLM:
         - Evaluar al menos 2 modelos con el mismo benchmark
         - Registrar: precisión, latencia media, coste estimado por consulta

[ ] 3.4  Análisis de errores: categorizar los fallos más frecuentes y proponer
         mejoras en los few-shots o en el pipeline.
